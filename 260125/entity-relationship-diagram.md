自己リレーション：自己リレーションとは"同じテーブル内でレコード同士を関連付けること"。つまり親と子供、上司と部下などのように親アカウントに対して子アカウントで同じテーブル内で管理される時に使用される。
経緯：今回仕様上、下記の例のような形を取るので中間テーブルを使用せずにusersで親と子供を一括管理する（この際roleカラムで親と子供を区別できるようにしておく）
例）　usersテーブルで1人の親に複数の子供が紐付いている場合👇
users

id	name	parent_id（親のid）
1	親A	NULL
2	子X	親Aの子（親のidを使用してログインする）
3	子Y	親Aの子（親のidを使用してログインする）
自己リレーションを使用できる理由：親は parent_id = null、子は parent_id = 親のid👉「親・子どもセット」で管理できるので別テーブルを作らずに関係性を表現可能。
無理なパターン：親が複数に対して子供も複数、親もルーティンを実行する場合、子供が家族外の複数のグループに所属している

選定理由：共通の項目が複数あり、中間テーブルが増えてJOINの回数が増えてパフォーマンス低下を防ぐため。シンプルなER図にして開発側の管理をしやすくするため。
懸念事項：複数の両親（お父さんとお母さんなど）で複数の子供を管理する場合はrole分けで判断できなくなってくるので中間テーブルが必要になってくる👉今後の親1人で管理していくという設計にするため今回は無視。その代わり共有の手段としてお父さんとお母さんが共有機能でURLとかで共有できたらいいかなって思ってる。
